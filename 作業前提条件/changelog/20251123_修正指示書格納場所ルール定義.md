# 修正指示書格納場所ルール定義

**作成日**: 2025年11月23日
**作成者**: Maintainer
**対象**: Worker, Reviewer, Maintainer全役割
**優先度**: 高
**ステータス**: 提案

## 背景

修正指示書の格納場所が作業前提条件ドキュメントで明確に定義されていないため、運用上の混乱が生じています。
本ドキュメントでは、修正指示書の種類別に適切な格納場所を定義します。

## 修正指示書の種類と格納場所

### 1. 作業前提条件・ルールに関する修正指示書

**作成者**: Maintainer
**内容**: 作業前提条件の変更、ルール改訂、フォーマット変更など
**格納場所**: `DOCS/作業前提条件/changelog/`
**ファイル名**: `YYYYMMDD_修正指示_[内容].md`

**例**:
- `20251123_修正指示_作業記録簡素化.md`
- `20251125_修正指示_命名規則改訂.md`

### 2. コードレビューによる修正指示書

**作成者**: Reviewer
**内容**: コードの品質改善、バグ修正、セキュリティ問題対応など
**格納場所**: 日付フォルダ（`YYYYMMDD/`）
**ファイル名**: `YYYYMMDD_修正指示_[対象名].md`

**例**:
- `20251123/20251123_修正指示_script.md`
- `20251124/20251124_修正指示_API改善.md`

### 3. プロジェクト固有の改善指示書

**作成者**: Maintainer（稀にReviewer）
**内容**: 特定プロジェクトのアーキテクチャ変更、大規模リファクタリング、機能改善など
**格納場所**: 各プロジェクトフォルダ内の `修正指示/` サブフォルダ
**ファイル名**: `YYYYMMDD_修正指示_[内容].md`

**例**:
- `python-satellite/修正指示/20251123_修正指示_API改善.md`
- `database_definition/修正指示/20251125_修正指示_DB最適化.md`
- `batch_file/修正指示/20251126_修正指示_スクリプト統合.md`

## Workerが修正指示書を探す際のフロー

```
1. まず日付フォルダを確認（日々のコード修正）
   → YYYYMMDD/内の修正指示書

2. 作業前提条件の変更が必要な場合
   → DOCS/作業前提条件/changelog/

3. プロジェクト固有の改善が必要な場合
   → 各プロジェクトフォルダ/修正指示/
```

## 各役割の責任

### Worker
- 修正指示書を確認し、指示に従って作業を実施
- 実施完了後、作業ログに記録

### Reviewer
- コードレビュー後、日付フォルダに修正指示書を作成
- Workerに修正指示書の存在を通知

### Maintainer
- 作業前提条件の改善指示は`DOCS/作業前提条件/changelog/`に作成
- プロジェクト全体の改善指示は`DOCS/`に作成
- 定期的に修正指示書の整理・アーカイブを実施

## 実装手順

1. 各作業前提条件ドキュメントにこのルールを追記
   - 作業前提条件_Worker_v2.8.md
   - 作業前提条件_Reviewer_v1.6.md
   - 作業前提条件_Maintainer_v1.3.md

2. 既存の修正指示書を適切な場所に移動（必要に応じて）

3. 今後の運用で徹底
