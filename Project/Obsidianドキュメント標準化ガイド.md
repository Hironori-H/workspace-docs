# Obsidianドキュメント標準化ガイド

---

## 📋 概要

このガイドは、Obsidian内の全ドキュメントを統一されたフォーマット・ガイドラインに準拠させるための標準化手順を定めたものです。

### 目的

1. **検索性の向上** - Frontmatterのタグで横断検索を可能に
2. **一貫性の確保** - すべてのドキュメントが同じ構造を持つ
3. **保守性の向上** - 更新履歴・ステータスの一元管理
4. **視覚性の向上** - Calloutsで重要情報を明確化

---

## 📖 ドキュメントタイプ（3種類）

### Type 1: 技術ガイド（詳細版）

**用途**: 手順書、設定ガイド、チュートリアル、詳細な技術ナレッジ

**判定基準**:
- 手順書、設定ガイド、詳細な説明がある


### リファレンス（40_Reference）

```yaml

例:

```

### プロジェクト（10_Projects）

```yaml
tags: [プロジェクト, プロジェクト名, 技術スタック1, 技術スタック2, ...]

例:

```

### README

```yaml
tags: [README, フォルダ名]

例:
tags: [README, Tech]
tags: [README, Projects]
tags: [README, Reference]
```

## 📝 Frontmatter設定

### 必須項目

```yaml
tags: [カテゴリタグ1, カテゴリタグ2, ...]
created: YYYY-MM-DD    # 作成日
updated: YYYY-MM-DD    # 更新日
type: guide/reference/note
```

### 任意項目

```yaml
aliases: [別名1, 別名2]  # Obsidian検索用別名
status: active/draft/archived  # ドキュメントのステータス
```

### 日付管理ルール

#### created（作成日）
- **既存のファイルに作成日の記載がある場合**: その日付を使用
- **作成日の記載がない場合**: ファイルのタイムスタンプやコンテキストから慎重に推定
- **推定できない場合**: 控えめな日付（例: `2025-11-01`）を設定

#### updated（更新日）
- **必ず作業日の日付に設定**
- created と updated は**絶対に同一にしない**

## 🔧 構造整備ガイド

### Type 1（技術ガイド）の整備項目

1. **概要セクション**をCallout `[!info]`で追加
   ```markdown
   > [!info] 概要
   > このドキュメントの目的と内容を1-2行で説明
   ```

2. **目次**を内部リンク `[[#セクション名]]` で作成
   ```markdown
   ## 📋 目次

   1. [[#セクション1]]
   2. [[#セクション2]]
   3. [[#関連情報]]
   ```

3. **セクション区切り** `---` を追加

4. **重要な推奨事項**を `[!tip]` で強調
   ```markdown
   > [!tip] ポイント
   > 重要な推奨事項
   ```

5. **注意事項**を `[!warning]` で強調
   ```markdown
   > [!warning] 注意
   > 注意すべき事項
   ```

6. **危険な操作**を `[!danger]` で強調
   ```markdown
   > [!danger] 危険
   > 重大な警告
   ```

7. **関連情報セクション**を追加
   ```markdown
   ## 関連情報

   ### 内部リンク
   - [[関連ノート1]]

   ### 外部リンク
   - [公式ドキュメント](URL)
   ```

8. **更新履歴**を表形式で追加
   ```markdown
   ## 📝 更新履歴

   | 日付 | バージョン | 更新内容 |
   |:---|:---|:---|
   | YYYY-MM-DD | 2.0 | ガイドライン適用 |
   | YYYY-MM-DD | 1.0 | 初版作成 |
   ```

9. 末尾に `[!quote]` で最終更新情報
   ```markdown
   > [!quote] 最終更新
   > **Last Updated**: YYYY-MM-DD
   > **Status**: ✅ Active
   ```

## 🎨 Calloutsリファレンス

| Callout | 用途 | 色 | 使用例 |
|:---|:---|:---|:---|
| `[!info]` | 概要、補足情報 | 青 | ドキュメント冒頭の概要 |
| `[!tip]` | 推奨事項、ベストプラクティス | 緑 | おすすめの方法、効率的なやり方 |
| `[!warning]` | 注意事項 | オレンジ | 注意すべき点、よくあるミス |
| `[!danger]` | 重大な警告 | 赤 | 危険な操作、データ損失リスク |
| `[!success]` | 成功パターン | 緑 | 推奨される実装例 |
| `[!quote]` | 引用、最終更新情報 | グレー | 末尾の最終更新情報 |

**構文**:
```markdown
> [!info] タイトル
> 内容行1
> 内容行2
```

## 🔗 内部リンク整備

既存の内部リンクを確認し、Obsidian形式 `[[ノート名]]` に統一：

**修正前**:
```markdown
- Git_SourceTree_ナレッジ
- 関連: Git_VSCode_SourceTree使い分け_ナレッジ
```

**修正後**:
```markdown
- [[Git_SourceTree_ナレッジ]]
- [[Git_VSCode_SourceTree使い分け_ナレッジ]]
```

**内部リンクの種類**:
- `[[ノート名]]` - 基本形
- `[[ノート名|表示名]]` - 別名表示
- `[[#セクション名]]` - 同一ファイル内リンク

## ⚠️ 重要な注意事項

### 必ず守ること

1. **既存の内容・情報は絶対に削除しない**
   - 追加・整形のみ
   - 情報の損失は厳禁

2. **created と updated を同一にしない**
   - created: 元の作成日を維持
   - updated: 作業日の日付

3. **Frontmatterの構文を正確に**
   ```yaml
   ---
   tags: [tag1, tag2]  # 配列形式
   created: 2025-11-20  # YYYY-MM-DD形式
   updated: 2025-12-04
   ---
   ```

4. **内部リンクは `[[]]` 形式**
   - `[[ノート名]]`
   - `[[ノート名|表示名]]`
   - `[[#セクション名]]`

5. **Calloutsの構文**
   ```markdown
   > [!info] タイトル
   > 内容
   ```

### 禁止事項

- ✗ 既存情報の削除
- ✗ 内容の大幅な書き換え
- ✗ created と updated を同一にする
- ✗ タグの無秩序な追加（命名規則に従う）
- ✗ Frontmatterの不正な構文

## 💬 よくある質問

### Q1. タイプ判定に迷った場合は？
A. 以下の基準で判断：
- セクション数が3つ以上 → Type 1
- 主に表形式 → Type 2
- 1-2セクション程度 → Type 3

### Q2. 作成日が不明な場合は？
A. ファイルのタイムスタンプやコンテキストから推定。推定できない場合は控えめな日付（例: `2025-11-01`）を設定。

### Q3. 既存の構造を大幅に変更したい場合は？
A. 基本は既存構造を保持。大幅な変更が必要な場合は、Maintainerに相談。

### Q4. status フィールドはいつ設定する？
A. 以下の場合に設定：
- `active`: 現在活用中のドキュメント
- `draft`: 作成中・未完成のドキュメント
- `archived`: 完了したプロジェクト、非推奨の情報

### Q5. aliases はどう使う？
A. Obsidianの検索機能で別名でも検索できるようにするため。例：
```yaml
aliases: [Claude Code, クロードコード]
```
